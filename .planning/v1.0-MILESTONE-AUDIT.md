## Gaps (Blocking)

### Gap 1 — MEMO-02: Medium-term memory read path missing

**Requirement:** MEMO-02 — System summarizes old conversations into episode summaries (medium-term memory)

**What works:** `scheduler/tasks/embedding.py` `summarize_episode.delay()` fires when conversation length exceeds `memory.episode_turn_threshold` (seed value from migration 007). `embed_and_store` with `entry_type="episode"` inserts rows into `memory_episodes` table with `vector(1024)` embeddings.

**What's missing:** `memory/medium_term.py` exports `load_recent_episodes(user_id, conversation_id)` — but this function is **never called** from `backend/agents/master_agent.py` `_load_memory_node`. The agent loads short-term turns and long-term facts but not episode summaries.

**Impact:** Episode summaries accumulate in the DB but are never surfaced to the agent. Medium-term memory is a write-only silo. An agent cannot "remember" conversation context from multiple sessions ago even if those sessions were summarized.

**Fix required:**
```python
# In backend/agents/master_agent.py _load_memory_node(), after loading short-term turns:
from memory.medium_term import load_recent_episodes
episodes = await load_recent_episodes(user_id=user_id, conversation_id=conversation_id)
# Prepend episode context to system message or state
```

**Location:** `backend/agents/master_agent.py` `_load_memory_node` (line ~118)

---

### Gap 2 — INTG-01: MCP server hot-registration not implemented

**Requirement:** INTG-01 — MCP servers connect via HTTP+SSE transport and are registered in the tool registry

**What works:** `MCPToolRegistry.refresh()` is called at startup in `backend/main.py` lifespan event. All servers in the DB at startup are discovered, their tools fetched via `list_tools`, and pre-registered in `gateway/tool_registry.py`.

**What's missing:** `backend/api/routes/mcp_servers.py` `create_mcp_server()` saves the new server to DB and commits, but does **not** call `MCPToolRegistry.refresh()`. Tools from a newly added server are not callable via `POST /api/tools/call` until the backend restarts.

**Impact:** The Admin → Settings → Integrations UI flow of adding a new MCP server does not result in that server's tools being immediately available. This defeats the purpose of the runtime admin registration.

**Fix required:**
```python
# In backend/api/routes/mcp_servers.py create_mcp_server(), after commit:
from mcp.registry import MCPToolRegistry
await MCPToolRegistry.refresh()
```

**Location:** `backend/api/routes/mcp_servers.py` line ~91 (after `await session.commit()`)

## Integration Findings (Non-blocking)

### Orphaned Export: `save_fact()` in `memory/long_term.py`

The `save_fact()` function exists and is well-documented, but `embed_and_store` Celery task inserts `MemoryFact` rows directly via ORM — `save_fact()` is never called in production paths. Functionally equivalent. Creates dead code confusion.

### Policy Deviation: BGE_M3Provider in FastAPI context

`backend/agents/master_agent.py` `_load_memory_node` instantiates `BGE_M3Provider()` inside the streaming response generator (FastAPI context). CLAUDE.md specifies bge-m3 must run in Celery workers only. The implementation uses `loop.run_in_executor(None, ...)` to avoid blocking the event loop. This is safe at 100-user scale but violates the stated architecture policy. Under high concurrency, the thread pool can saturate.

## Security Gate Assessment

The 3-gate security chain holds across all code paths:
- **Gate 1 (JWT):** All routes use `Depends(get_current_user)` — zero unprotected sensitive endpoints
- **Gate 2 (RBAC):** All agent + MCP + admin routes check `has_permission()`
- **Gate 3 (ACL):** All tool calls check `check_tool_acl()` against `tool_acl` table
- **Audit logging:** `log_tool_call()` fires on every tool call (allow and deny) — no credentials in logs
- **Memory isolation:** All memory queries parameterized on `user_id` from JWT — zero cross-user read paths

The security architecture is sound and ready for production.
