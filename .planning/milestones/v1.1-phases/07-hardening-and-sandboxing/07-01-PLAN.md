  <behavior>
    SandboxExecutor.execute(code: str, language: str, timeout: int = 30) -> SandboxResult

    SandboxResult fields:
    - stdout: str
    - stderr: str
    - exit_code: int
    - timed_out: bool
    - container_id: str | None

    Resource constraints enforced via Docker SDK:
    - CPU: nano_cpus = 500_000_000 (0.5 cores = 500M nanocpus)
    - Memory: mem_limit = "256m"
    - Network: network_disabled = True (NOT network_mode="none" which allows localhost)
    - Filesystem: read_only=True; tmpfs={"/tmp": "size=64m,mode=777"}; no host volume mounts
    - Container image: python:3.12-alpine for Python execution
    - Container destroyed immediately after run (auto_remove=True OR explicit remove())

    Timeout enforcement:
    - containers.run() with timeout parameter from SandboxExecutor.execute()
    - On ContainerError or APIError with timeout cause: set timed_out=True, force-remove container
    - Leaked container cleanup: periodic _cleanup_leaked_containers() function using containers.list(filters={"label": "blitz.sandbox=true"})

    Cases (for TDD):
    - execute(code="print('hello')", language="python") → SandboxResult(stdout="hello\n", exit_code=0, timed_out=False)
    - execute(code="import os; os.system('ls /')", ...) → runs in container (isolated), exit_code=0 (command runs but sees container fs, not host)
    - execute(code="raise Exception('err')", ...) → SandboxResult(exit_code=1, stderr contains "Exception")
    - execute(code="while True: pass", timeout=1) → SandboxResult(timed_out=True)
    - execute after container destroyed: containers.list() returns no blitz.sandbox containers
  </behavior>
