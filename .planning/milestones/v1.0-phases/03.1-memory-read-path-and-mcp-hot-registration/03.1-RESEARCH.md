## Summary

Phase 3.1 closes two integration gaps found in the v1.0 milestone audit. Both gaps are surgical fixes to existing, correct infrastructure — no new modules, no new DB migrations, no new dependencies. The write side of both features is complete and tested; only the read/notification side is missing.

**Gap 1 (MEMO-02):** `load_recent_episodes()` exists in `memory/medium_term.py` and is fully implemented and tested. `_load_memory_node` in `master_agent.py` already loads short-term turns and long-term facts using the same pattern. The fix is to call `load_recent_episodes()` inside `_load_memory_node` and inject the episode summaries as a `SystemMessage` prefix — identical in structure to how long-term facts are injected today. The function signature matches what is needed: `load_recent_episodes(session, user_id=user_id, n=5)`. No schema change, no migration, no new table.

**Gap 2 (INTG-01):** `MCPToolRegistry.refresh()` is an idempotent class method that re-reads the DB, reconnects all active servers, and re-registers their tools. It is already called at startup. The fix is to call it again inside `create_mcp_server()` after `await session.commit()`. The call is `await MCPToolRegistry.refresh()` — one line. The existing `_clients` dict and `_registry` in `tool_registry.py` are module-level singletons that survive the lifetime of the FastAPI process, so re-calling `refresh()` safely updates them in place.

**Gap 3 (stale test):** Already fixed. `test_create_master_graph_has_routing_conditional` now checks `load_memory` branches (not `master_agent`). All 177 tests pass as of 2026-02-26.

**Primary recommendation:** Implement both fixes as a single plan wave. Fix 1 requires one new test (episode injection into agent context). Fix 2 requires one new test (hot-registration round-trip). Both are unit tests using the existing mock-session pattern established in `test_master_agent_memory.py` and `test_mcp_registry.py`.

## Architecture Patterns

### Pattern 1: Episode Injection in `_load_memory_node`

**What:** After loading long-term facts via `search_facts()`, call `load_recent_episodes()` and prepend episode summaries as a second `SystemMessage` before the conversation history.

**When to use:** Always — the injection follows the same conditional pattern as long-term facts: only inject if episodes exist, wrap in try/except for graceful degradation.

**Current state of `_load_memory_node` (relevant excerpt):**

```python
# backend/agents/master_agent.py — _load_memory_node, after line ~142

# EXISTING: Long-term facts injection (already works)
if last_user_message and user_id:
    try:
        provider = BGE_M3Provider()
        query_embedding = (await provider.embed([str(last_user_message)]))[0]
        async with async_session() as session:
            facts = await search_facts(session, user_id=user_id, query_embedding=query_embedding, k=5)
        loaded_facts = [f.content for f in facts]
        if loaded_facts:
            facts_context = "\n".join(f"- {fact}" for fact in loaded_facts)
            history.insert(0, SystemMessage(content=f"[Long-term memory — relevant facts about this user:]\n{facts_context}"))
    except Exception:
        logger.warning("long_term_memory_load_failed", user_id=str(user_id))

# MISSING: Medium-term episode injection (the gap)
# Add AFTER long-term facts block:
if user_id:
    try:
        async with async_session() as session:
            episodes = await load_recent_episodes(session, user_id=user_id, n=5)
        if episodes:
            episode_context = "\n".join(f"- {ep.summary}" for ep in episodes)
            history.insert(
                0,
                SystemMessage(
                    content=f"[Medium-term memory — summaries of past conversations:]\n{episode_context}"
                ),
            )
            logger.debug("medium_term_memory_loaded", episode_count=len(episodes), user_id=str(user_id))
    except Exception:
        logger.warning("medium_term_memory_load_failed", user_id=str(user_id))
```

**Import to add at top of `master_agent.py`:**
```python
from memory.medium_term import load_recent_episodes
```

**Position matters:** Episode summaries should be inserted at position 0 (before long-term facts) or at position 1 (after long-term facts) — recommendation is to insert at 0 so the LLM sees the oldest context first, then most-recent facts. If long-term facts are already at 0, episodes should be inserted at 0 (pushing facts to 1). This preserves the intuitive ordering: episode context (broader history) → facts (specific details) → conversation turns.

**Graceful degradation:** The try/except pattern is mandatory. `load_recent_episodes()` makes a DB call that can fail (connection pool exhaustion, DB unavailable). The agent must continue without episode context in that case — same as long-term facts.

### Pattern 2: MCP Hot-Registration After `create_mcp_server()`

**What:** After `await session.commit()` in `create_mcp_server()`, call `await MCPToolRegistry.refresh()` to immediately register the new server's tools in the in-process registry.

**Current state of `create_mcp_server()` (lines 70-103 of `api/routes/mcp_servers.py`):**

```python
@router.post("")
async def create_mcp_server(
    body: McpServerCreate,
    user: UserContext = Depends(_require_admin),
    session: AsyncSession = Depends(get_db),
) -> dict[str, Any]:
    """Register a new MCP server. Encrypts auth_token with AES-256-GCM before storage."""
    from security.credentials import encrypt_token
    # ... encrypt, create server, session.add(server) ...
    await session.commit()
    await session.refresh(server)
    logger.info("mcp_server_registered", name=body.name, user_id=str(user["user_id"]))

    # MISSING: Hot-register the new server's tools immediately
    # Add AFTER logger.info:
    # from mcp.registry import MCPToolRegistry
    # await MCPToolRegistry.refresh()

    return {"id": str(server.id), "name": server.name, "url": server.url, "is_active": server.is_active}
```

**Import pattern:** Import `MCPToolRegistry` inside the function (lazy import) to avoid circular imports — the same pattern used in `MCPToolRegistry.refresh()` itself which imports `core.db` and `core.models.mcp_server` lazily.

**Idempotency:** `refresh()` re-queries all active servers from DB. Adding a new server and then calling `refresh()` will include the new server in the query — this is correct. For servers that were already registered, `register_tool()` will overwrite with identical data (idempotent). For the new server, it will call `list_tools()` on the new server's SSE endpoint and register its tools.

**Error handling in `refresh()`:** `refresh()` already wraps each server connection in try/except and logs `mcp_server_unavailable` on failure. If the new server is unreachable at registration time, `refresh()` will log the warning but not raise — the `create_mcp_server()` endpoint will still return 200. This is correct: the server is saved to DB and will be picked up on the next restart or next `refresh()` call.

**Race condition:** If two admins simultaneously call `POST /api/admin/mcp-servers`, both will call `refresh()`. Since `refresh()` reads from DB and re-registers, both calls are safe — the last write wins for the same server name in `_registry` and `_clients`, but since they're registering different servers, there is no conflict.

### Anti-Patterns to Avoid

- **Don't add a new `loaded_episodes` field to `BlitzState`:** The audit trail purpose is served by the structlog debug log. Adding a state field risks LangGraph state schema issues and is unnecessary complexity. Only `loaded_facts` is in state (for audit); episodes can be logged.
- **Don't make `refresh()` call blocking the HTTP response:** `await MCPToolRegistry.refresh()` is async and takes some time (one `list_tools` HTTP call per active server). If there are many servers or a server is slow, this adds latency to the `create_mcp_server` response. This is acceptable — admin registration is a rare operation, not on the hot path.
- **Don't wrap `await MCPToolRegistry.refresh()` in try/except that swallows errors silently:** If `refresh()` itself raises (which it won't — it catches per-server errors internally), the error should propagate or be logged at the route level.
- **Don't call `load_recent_episodes()` inside the `if last_user_message and user_id:` guard:** Episodes are user-scoped, not query-scoped — they don't need the last user message or embedding. They should be loaded unconditionally when `user_id` is available.

## Common Pitfalls

### Pitfall 1: Import Position for `load_recent_episodes`

**What goes wrong:** Adding `from memory.medium_term import load_recent_episodes` inside `_load_memory_node` (lazy import) instead of at the top of `master_agent.py`.

**Why it happens:** Lazy imports seem safe but `patch("agents.master_agent.load_recent_episodes")` will fail if the name is not module-level — the same bug that caused the `call_mcp_tool` import issue in Phase 03-04.

**How to avoid:** Add the import at the top of `master_agent.py` alongside `from memory.long_term import search_facts` and `from memory.short_term import load_recent_turns`.

**Warning signs:** Test patch `patch("agents.master_agent.load_recent_episodes")` raises `AttributeError: <module 'agents.master_agent'> does not have attribute 'load_recent_episodes'`.

**Reference decision:** `[Phase 03-04]: call_mcp_tool imported at top level in project_agent.py (not lazily) — lazy import not patchable in tests`

### Pitfall 2: `load_recent_episodes` Signature vs. Current Call Pattern

**What goes wrong:** Calling `load_recent_episodes(session, user_id=user_id, conversation_id=conversation_id)` — the function does NOT have a `conversation_id` parameter. It is user-scoped, not conversation-scoped.

**Why it happens:** The function signature in `medium_term.py` is `load_recent_episodes(session, *, user_id, n=5)` — no `conversation_id`. Episode summaries are deliberately user-scoped (cross-conversation context). The audit correctly states "5 most recent episodes for a user."

**How to avoid:** Read the function signature before calling. Confirmed correct call: `await load_recent_episodes(session, user_id=user_id, n=5)`.

### Pitfall 3: `MCPToolRegistry.refresh()` is a classmethod, not an instance method

**What goes wrong:** Creating an instance: `registry = MCPToolRegistry(); await registry.refresh()`.

**Why it happens:** The class is designed as a namespace with a single classmethod — `@classmethod async def refresh(cls)`. It has no `__init__`.

**How to avoid:** Always call as `await MCPToolRegistry.refresh()` (class-level call).

### Pitfall 4: Session Already Committed When `refresh()` Queries DB

**What goes wrong:** Assuming the session used in `create_mcp_server()` is still usable after `await session.commit()`. `refresh()` opens its own session via `async with async_session() as session:` internally — it does not use the route's session. There is no conflict.

**How to avoid:** No action needed — `refresh()` manages its own session independently.

### Pitfall 5: Episode injection position in history list

**What goes wrong:** Inserting episode `SystemMessage` at position 0 when long-term facts are already at position 0, resulting in episodes appearing after facts in the history (counterintuitive).

**Why it happens:** Both inject at `history.insert(0, ...)` — the second call pushes the first call's message to position 1.

**How to avoid:** Insert episodes at position 0 (broader context first, then specific facts). Long-term facts are already inserted at 0. Since episodes are inserted after facts (episodes block comes after the facts block in the code), the ordering will be: episodes at 0, facts at 1, conversation history at 2+. This is the correct semantic order — broadest context first.

## Open Questions

1. **Episode injection position relative to long-term facts**
   - What we know: Both long-term facts and episodes are injected as `SystemMessage` at `history.insert(0, ...)`. The second call will push the first to position 1.
   - What's unclear: Should episodes appear before or after long-term facts semantically? Episodes are broader/older context; facts are specific. Broader before specific is conventional.
   - Recommendation: Insert episodes FIRST (before the long-term facts block in code), so episodes end up at position 1 and facts at position 0. This means facts are closest to the conversation (position 0), which is semantically correct — the most specific relevant context is nearest. OR: insert episodes at the END of history as a separate block (not at position 0). This avoids ordering confusion entirely. **Recommended: insert at 0 in a block that runs BEFORE the long-term facts block, so final order is: facts (0), episodes (1), conversation (2+).**

2. **Should `refresh()` failure in `create_mcp_server()` be surfaced to the caller?**
   - What we know: `refresh()` internally catches per-server errors. If the NEW server is unreachable, `refresh()` logs `mcp_server_unavailable` but does not raise.
   - What's unclear: Should the route return a 207 Multi-Status or 200 with a warning field if `refresh()` could not connect to the new server?
   - Recommendation: Return 200 always. The DB record is committed (server is registered). The warning is in server logs. An admin can check server connectivity separately. Over-engineering the response for a rare edge case violates KISS.

3. **n=5 for episodes — is this the right default?**
   - What we know: `load_recent_episodes(session, user_id=user_id, n=5)` is already the function default. Long-term facts use `k=5` for semantic search.
   - What's unclear: At 100-user scale with frequent use, 5 episodes may not be enough context. But more episodes = more tokens.
   - Recommendation: Use n=3 for the initial implementation (less context pressure than facts which are specific). Document it as configurable later if needed. The exact value is low-stakes — the important thing is the plumbing works.
