  <action>
    Create admin CRUD routes following the existing pattern from `api/routes/system_config.py` (dependency override for auth, SQLite test fixture).

    **admin_agents.py** (prefix `/api/admin/agents`):
    - Shared dependency `_require_registry_manager` that calls `get_current_user` + checks `registry:manage` via `has_permission`. Returns 403 if denied.
    - `GET /` -- list all agent definitions, optional `?status=active` and `?version=1.0.0` filters. Returns list with is_active flag.
    - `POST /` -- create new agent definition from AgentDefinitionCreate body, returns 201. Version defaults to "1.0.0". is_active defaults to True.
    - `GET /{agent_id}` -- get by UUID, returns 404 if not found
    - `PUT /{agent_id}` -- update fields from AgentDefinitionUpdate body
    - `PATCH /{agent_id}/status` -- enable/disable from StatusUpdate body. **Graceful removal (EXTD-05)**: When setting status to "disabled" or "deprecated", query workflow_runs table for active runs referencing this agent (JOIN via workflow.definition_json agent references or agent_definitions.name). Return response with `{"updated": true, "active_workflow_runs": N}` so admin can see impact.
    - `PATCH /{agent_id}/activate` -- set is_active=True on this version, set is_active=False on all other versions of the same agent name. Enables version rollback.
    - `PATCH /bulk-status` -- accept BulkStatusUpdate body (list of IDs + status), update all matching rows. Return count updated.
    - All endpoints use `get_db` for async session
    - Audit log on create/update/status-change via `structlog.get_logger(__name__)`

    **admin_tools.py** (prefix `/api/admin/tools`):
    - Same pattern as admin_agents.py but for ToolDefinition model and ToolDefinitionCreate/Update/Response schemas
    - Same endpoints: list (with version filter), create, get, update, patch status (with active workflow count), activate version, bulk-status
    - Same `_require_registry_manager` dependency pattern

    **Tests for both:**
    - Use TestClient + SQLite in-memory DB fixture (same pattern as test_system_config.py)
    - Override `get_current_user` dependency to inject mock user with `roles=["it-admin"]`
    - Override `get_db` to use SQLite async session
    - Test cases per route: 401 without auth, 403 with employee role, 200/201 with admin role
    - Test CRUD flow: create -> get -> list -> update -> patch status -> verify
    - Test multi-version: create v1.0.0, create v1.1.0, activate v1.1.0, verify v1.0.0 is_active=False
    - Test bulk-status: create 3 items, bulk disable 2, verify
    - Test graceful removal: status patch returns active_workflow_runs count

    **Update main.py:**
    - Import and register both routers: `app.include_router(admin_agents.router)` and `app.include_router(admin_tools.router)`

    **admin_skills.py** (prefix `/api/admin/skills`):
    - Same `_require_registry_manager` dependency pattern
    - `GET /` -- list all skill definitions, optional `?status=active`, `?skill_type=procedural`, `?version=1.0.0` filters
    - `POST /` -- create skill from SkillDefinitionCreate. Pydantic schema validates skill_type vs instruction_markdown/procedure_json consistency
    - `GET /pending` -- list skills with `status='pending_review'` (for admin review queue). Declare BEFORE `/{skill_id}` to avoid FastAPI UUID matching.
    - `GET /{skill_id}` -- get by UUID
    - `PUT /{skill_id}` -- update from SkillDefinitionUpdate
    - `PATCH /{skill_id}/status` -- enable/disable. Returns active_workflow_runs count (EXTD-05 graceful removal).
    - `PATCH /{skill_id}/activate` -- activate specific version (same multi-version logic as agents)
    - `PATCH /bulk-status` -- bulk status update
    - `POST /{skill_id}/validate` -- dry-run: load procedure_json and validate structure (tool refs exist, variable refs resolve, step count <= 20). Returns list of errors or empty (valid). Uses SkillValidator from Plan 06-05 -- stub for now, return empty list.
    - Import and review endpoints are stubs returning 501 -- implemented in Plan 06-05 (skill import pipeline).

    **admin_permissions.py** (prefix `/api/admin/permissions`):

    **Role permissions:**
    - `GET /roles` -- return all role-permission mappings grouped by role. Query role_permissions table, return dict[str, list[str]].
    - `PUT /roles/{role}` -- replace all permissions for a role. Delete existing rows for role, insert new rows from RolePermissionSet body. Call `invalidate_permission_cache()` after write.

    **Artifact permissions (staged model per user decision):**
    - `GET /artifacts/{artifact_type}/{artifact_id}` -- return artifact permission overrides (both active and pending). Query artifact_permissions WHERE artifact_type and artifact_id, return list of {role, allowed, status}.
    - `PUT /artifacts/{artifact_type}/{artifact_id}` -- set per-role permissions for artifact. Delete existing rows, insert new rows from ArtifactPermissionSet body **with status='pending'** (staged model). Does NOT call invalidate_permission_cache yet.
    - `POST /apply` -- accept PermissionApplyRequest (list of permission IDs). Update matching artifact_permissions rows from status='pending' to status='active'. Call `invalidate_permission_cache()` after applying. Returns count of applied permissions.

    **Per-user overrides:**
    - `GET /users/{artifact_type}/{artifact_id}` -- return user-level permission overrides for an artifact.
    - `PUT /users/{artifact_type}/{artifact_id}` -- set per-user permissions. Insert with status='pending' (staged). Accepts list of {user_id, allowed}.
    - (Per-user overrides are also applied via POST /apply)

    **Tests:**
    - Same pattern: TestClient + SQLite + dependency overrides
    - test_admin_skills.py: full CRUD flow, test validate endpoint, test pending filter, test multi-version, test bulk-status
    - test_admin_permissions.py: test role listing, test setting permissions creates pending rows, test apply makes them active, test invalidation only on apply, test per-user override CRUD, test artifact permission CRUD
    - Test staged model: write permission -> verify status='pending' -> apply -> verify status='active' -> verify cache invalidated

    **Update main.py:**
    - Register both new routers
