<decisions>
## Implementation Decisions

### Channel Delivery Failure Handling
- Use existing 3x exponential backoff (send_outbound) for delivery retries; if all retries fail, mark the workflow run as failed
- If owner has no linked channel account for the target channel type, fail the workflow run with clear error (e.g. "No linked Telegram account for user X")
- Store node-level error detail in WorkflowRun: failing node ID + error message so users can debug their workflows
- On successful delivery, log delivery metadata (channel type, external_chat_id, delivery timestamp) in the workflow run output for audit

### Agent Node Output Format
- Use formatted text output — reuse the existing `_format_for_channel()` logic from `gateway.py` to convert structured sub-agent data to human-readable text
- Wire all three sub-agents: email, calendar, and project (all implemented and tested from Phase 3)
- If a sub-agent call fails (LLM timeout, tool error), fail the workflow with node-level error — consistent with channel delivery failure behavior
- Use the agent_node's prompt/instructions field to guide the sub-agent invocation (pass as user query to the sub-agent)

### Role Resolution Strategy
- Fetch owner's Keycloak roles via Keycloak Admin API (existing keycloak_client.py) per workflow execution — always fresh, no caching
- If Keycloak is unreachable, fail the workflow run — security-first: no roles = no tool access = workflow can't function
- Log the resolved role list in workflow run output metadata for audit and debugging ACL issues

### Multi-Channel Targeting
- One channel per channel_output_node — users add multiple nodes for multi-channel delivery (simple, composable)
- Channel type specified as a `channel_type` field in node data properties (e.g. 'telegram', 'whatsapp', 'teams')
- Resolve external_chat_id at runtime (not compile time) — look up from channel_accounts table using owner_user_id + channel_type when node executes
- Prefix delivered messages with workflow name (e.g. "[Morning Digest] ...") so users know which workflow sent the message

### Claude's Discretion
- Exact implementation of node-level error storage in WorkflowRun (new column vs JSON field in existing output)
- How to structure the Keycloak Admin API call in keycloak_client.py (new method or extend existing)
- How to pass workflow name context to the delivery function

</decisions>
