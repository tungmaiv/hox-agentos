## Summary

This phase closes three integration gaps identified in the v1.1 milestone audit. All three are backend-only changes with no new dependencies, no new database tables, and no frontend modifications. The work is connecting existing subsystems that were built in separate phases (Phase 4 workflow engine + Phase 5 channel delivery).

**Gap 1 -- Channel delivery:** `_handle_channel_output_node` in `node_handlers.py` builds an `InternalMessage` with `external_user_id=""` because the workflow's `user_context` dict has no channel identity. The fix is to query `channel_accounts` table at runtime using `owner_user_id + channel_type` to resolve the `external_user_id` (which doubles as the `external_chat_id` for direct message delivery).

**Gap 2 -- Agent node stub:** `_handle_agent_node` returns hardcoded `{"agent": "...", "result": "[stub] ...", "success": True}`. The fix is to invoke the real sub-agent functions (`email_agent_node`, `calendar_agent_node`, `project_agent_node`) and format their output with `_format_for_channel()` (already in `channels/gateway.py`).

**Gap 3 -- Worker roles:** `workflow_execution.py` line 93 reads `owner_roles_json` from the WorkflowRun, but `cron_trigger.py` line 74 copies `trigger.owner_roles_json` which was set at trigger creation time. If a user's roles change after trigger creation, workflows execute with stale roles. The fix is to query Keycloak Admin API for fresh roles at execution time.

**Primary recommendation:** Implement all three fixes in `node_handlers.py` and `workflow_execution.py` with a new `security/keycloak_client.py` module for role fetching. No schema migrations needed -- node-level errors can be stored in the existing `result_json` JSONB column on `WorkflowRun`.

### Pattern 1: Runtime Channel Account Resolution

**What:** Query `channel_accounts` table at workflow execution time to resolve the delivery target.
**When to use:** When `channel_output_node` executes and needs `external_user_id` + `external_chat_id`.

The `channel_accounts` table has:
- `user_id` (UUID) -- the Blitz user ID (from Keycloak/JWT)
- `channel` (String) -- "telegram", "whatsapp", "ms_teams"
- `external_user_id` (String) -- the platform-specific user ID (e.g., Telegram chat ID)
- `is_paired` (Boolean) -- must be True for delivery

Resolution query:
```python
# Source: backend/core/models/channel.py (ChannelAccount model)
from sqlalchemy import and_, select
from core.models.channel import ChannelAccount
from core.db import async_session

async def resolve_channel_account(
    owner_user_id: uuid.UUID,
    channel_type: str,
) -> ChannelAccount | None:
    async with async_session() as session:
        result = await session.execute(
            select(ChannelAccount).where(
                and_(
                    ChannelAccount.user_id == owner_user_id,
                    ChannelAccount.channel == channel_type,
                    ChannelAccount.is_paired == True,  # noqa: E712
                )
            )
        )
        return result.scalar_one_or_none()
```

**Key insight:** For Telegram, `external_user_id` IS the chat ID for direct messages. The `ChannelSession.external_chat_id` is set to `msg.external_chat_id or msg.external_user_id` in `gateway.py:66-67`. For workflow-initiated outbound delivery, we use `external_user_id` as both fields since workflows send DMs to the owner.

### Pattern 2: Sub-Agent Invocation in Workflow Context

**What:** Call the existing sub-agent functions from `_handle_agent_node` instead of returning stubs.
**When to use:** When an `agent_node` in a workflow needs real LLM output.

The sub-agents are designed for the conversational graph (BlitzState with messages), not for the workflow graph (WorkflowState). The approach is:

```python
# Sub-agents expect BlitzState with messages, return {"messages": [AIMessage(...)]}
# Workflow agent_node receives config with "agent" and "instruction" fields.
# Bridge: create a minimal BlitzState with a HumanMessage from the instruction,
# invoke the sub-agent, extract the AIMessage content, format for channel.

from langchain_core.messages import HumanMessage, AIMessage

async def _handle_agent_node(config: dict, state: WorkflowState) -> Any:
    agent_name = config.get("agent", "email_agent")
    instruction = config.get("instruction", "")

    # Build minimal BlitzState for sub-agent
    mini_state: BlitzState = {
        "messages": [HumanMessage(content=instruction)],
        # ... other required fields
    }

    # Dispatch to correct sub-agent
    agent_fn = AGENT_DISPATCH[agent_name]
    result = await agent_fn(mini_state)

    # Extract AIMessage content
    messages = result.get("messages", [])
    ai_content = next(
        (str(m.content) for m in messages if isinstance(m, AIMessage)),
        f"No response from {agent_name}"
    )

    # Format for channel delivery (reuse existing formatter)
    formatted = _format_for_channel(ai_content)
    return {"agent": agent_name, "result": formatted, "success": True}
```

### Pattern 3: Keycloak Admin API Service Account Authentication

**What:** Use the backend's `keycloak_client_secret` (client credentials grant) to get an admin token, then call the role mappings endpoint.
**When to use:** In Celery workers before workflow execution to fetch the owner's current roles.

The Keycloak Admin REST API endpoint is:
```
GET /admin/realms/{realm}/users/{user-id}/role-mappings/realm
```

Authentication flow:
1. POST to `/realms/{realm}/protocol/openid-connect/token` with `grant_type=client_credentials`, `client_id`, `client_secret`
2. Use the returned `access_token` as `Authorization: Bearer <token>` for the admin API call
3. Parse the role list from the JSON response

```python
# Source: Keycloak REST API docs (Context7 verified)
async def fetch_user_roles(user_id: str) -> list[str]:
    settings = get_settings()
    token_url = f"{settings.keycloak_url}/realms/{settings.keycloak_realm}/protocol/openid-connect/token"

    # Step 1: Get admin token via client credentials
    async with httpx.AsyncClient(verify=settings.keycloak_ca_cert or True) as client:
        token_resp = await client.post(token_url, data={
            "grant_type": "client_credentials",
            "client_id": settings.keycloak_client_id,
            "client_secret": settings.keycloak_client_secret,
        })
        token_resp.raise_for_status()
        admin_token = token_resp.json()["access_token"]

        # Step 2: Fetch realm role mappings
        roles_url = (
            f"{settings.keycloak_url}/admin/realms/{settings.keycloak_realm}"
            f"/users/{user_id}/role-mappings/realm"
        )
        roles_resp = await client.get(
            roles_url,
            headers={"Authorization": f"Bearer {admin_token}"},
        )
        roles_resp.raise_for_status()
        return [r["name"] for r in roles_resp.json()]
```

**CRITICAL:** The `keycloak_client_id` (blitz-backend) must have a service account with the `realm-management` client role `view-users` assigned. This is a Keycloak admin setup step, not a code change. If the service account lacks this role, the API returns 403.

### Pattern 4: Node-Level Error Storage

**What:** Store per-node error details in the existing `WorkflowRun.result_json` JSONB column.
**When to use:** When any node fails during workflow execution.

Recommendation: Use `result_json` (already JSONB, already exists) with a structured error format:

```python
run.result_json = {
    "error": str(exc),
    "failed_node_id": node_id,
    "failed_node_type": node_type,
}
```

This avoids a new column and migration. The existing error path already writes `run.result_json = {"error": str(exc)}` -- we extend it with `failed_node_id`.

To capture the failing node ID during execution, modify the `_make_node_fn` closure in `graphs.py` to catch exceptions and re-raise with node context, or handle in the `workflow_execution.py` event stream by tracking the currently executing node.

### Anti-Patterns to Avoid

- **Don't add external_user_id to WorkflowState:** The channel account lookup should be inside the node handler, not passed through the entire graph state. WorkflowState is domain-agnostic.
- **Don't cache Keycloak roles in WorkflowRun.owner_roles_json:** The locked decision says "always fresh, no caching". The `owner_roles_json` column is still useful as a fallback and audit record, but the live fetch takes precedence.
- **Don't import the full master_agent graph for agent_node dispatch:** Import the individual sub-agent functions directly. The master graph includes memory nodes, delivery router, etc. that are not needed for workflow agent_node invocation.
- **Don't create a new DB column for node errors:** `result_json` is JSONB and can hold any structure. A new column means a migration for no benefit.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|

## Common Pitfalls

### Pitfall 1: external_user_id vs external_chat_id Confusion

**What goes wrong:** The `InternalMessage` model has both `external_user_id` and `external_chat_id`. Using the wrong one causes delivery failure.
**Why it happens:** For Telegram DMs, the "chat ID" is the same as the "user ID". For group chats, they differ. Workflows deliver to the user's DM, not a group.
**How to avoid:** Set both `external_user_id` AND `external_chat_id` to the value from `ChannelAccount.external_user_id`. The sidecar uses `external_chat_id` for the API call target. Looking at the Telegram sidecar: it uses `msg.external_chat_id` as the `chat_id` parameter in `send_message`.
**Warning signs:** `send_outbound` succeeds but message goes to wrong chat, or sidecar returns 400 because `external_chat_id` is empty.

### Pitfall 2: Sub-Agent BlitzState vs WorkflowState Type Mismatch

**What goes wrong:** Sub-agents expect `BlitzState` (TypedDict with `messages`, `user_id`, `conversation_id`, etc.) but workflow nodes have `WorkflowState` (TypedDict with `run_id`, `user_context`, `node_outputs`, etc.).
**Why it happens:** Sub-agents were built for the conversational graph, not the workflow graph.
**How to avoid:** Create a minimal `BlitzState` dict inside `_handle_agent_node` with only the fields the sub-agent actually reads. Looking at the sub-agent code: `email_agent_node` only reads nothing from state (it logs "email_agent_invoked" and returns mock data). `calendar_agent_node` and `project_agent_node` follow the same pattern. They don't actually use `state` fields -- they return hardcoded mock data in Phase 3. So the BlitzState bridge is minimal: just `{"messages": [HumanMessage(content=instruction)]}`.
**Warning signs:** `TypeError` or `KeyError` when sub-agent tries to access state fields that don't exist in the minimal dict.

### Pitfall 3: Celery Worker Cannot Reach Keycloak

**What goes wrong:** Celery workers run in a separate process. The `KEYCLOAK_URL` setting uses Docker service name (`http://keycloak:8080`), which may not resolve if the worker runs on the host.
**Why it happens:** Backend runs on host (via `just backend`), but Keycloak runs in Docker.
**How to avoid:** Check that `KEYCLOAK_URL` in `backend/.env` is accessible from wherever the Celery worker runs. If running locally: use `https://keycloak.blitz.local:7443` (the external HTTPS address from MEMORY.md). If running in Docker: use the Docker service name. The locked decision says "if Keycloak is unreachable, fail the workflow run" -- implement this as a try/except with clear error.
**Warning signs:** `httpx.ConnectError` or `httpx.TimeoutException` during role fetch in Celery worker logs.

### Pitfall 4: Keycloak Service Account Lacks Admin Permissions

**What goes wrong:** The `keycloak_client_id` (blitz-backend) service account token cannot call the admin API because it lacks the `view-users` role in `realm-management`.
**Why it happens:** Keycloak's service accounts have no admin permissions by default. The admin role must be explicitly assigned.
**How to avoid:** Document the Keycloak admin setup step: assign `realm-management` > `view-users` role to the `blitz-backend` service account. Include a fallback: if role fetch fails, log a warning and fall back to `owner_roles_json` from the WorkflowRun (stale but better than nothing -- wait, the locked decision says "fail the workflow run"). So no fallback -- fail.
**Warning signs:** 403 from Keycloak admin API.

### Pitfall 5: _format_for_channel is an Instance Method

**What goes wrong:** `_format_for_channel` is defined as `self._format_for_channel(text)` on `ChannelGateway` class. Cannot call it without a gateway instance.
**Why it happens:** It was designed as part of the `_invoke_agent` flow which already has `self`.
**How to avoid:** Either: (a) make it a standalone function / static method, or (b) import `get_channel_gateway()` and call `gateway._format_for_channel()`. Option (b) is simpler since `get_channel_gateway()` is already imported in `_handle_channel_output_node`. However, for cleanliness, extracting it as a module-level function is better since it has no dependency on `self`.
**Warning signs:** `TypeError: _format_for_channel() missing 1 required positional argument: 'self'` if called without instance.

### Pitfall 6: Workflow Name Not Available in Node Handler

**What goes wrong:** The locked decision says "prefix delivered messages with workflow name". But `WorkflowState` does not contain the workflow name -- only `run_id` and `user_context`.
**Why it happens:** The workflow name lives in the `Workflow` model, which is loaded in `workflow_execution.py` but not passed into the graph state.
**How to avoid:** Add `workflow_name` to `WorkflowState` (or pass it in `user_context`). The cleanest approach: add a `workflow_name: str | None` field to `WorkflowState` and populate it in the initial_state dict in `workflow_execution.py`.
**Warning signs:** Channel messages arrive without the `[Workflow Name]` prefix because the handler can't access the name.

### Example 1: Fixed _handle_channel_output_node

```python
# Source: backend/agents/node_handlers.py (to be modified)
async def _handle_channel_output_node(config: dict[str, Any], state: WorkflowState) -> Any:
    channel = config.get("channel", "web")
    template = config.get("template", "{output}")
    output = state.get("current_output")
    user_context = state.get("user_context") or {}
    workflow_name = state.get("workflow_name", "")

    try:
        message = template.format(output=output)
    except (KeyError, ValueError):
        message = str(output)

    # Prefix with workflow name per locked decision
    if workflow_name:
        message = f"[{workflow_name}] {message}"

    if channel == "web":
        return {"channel": channel, "message": message, "sent": True}

    # Resolve delivery target from channel_accounts
    owner_user_id = user_context.get("user_id")
    if not owner_user_id:
        return {"error": "No user_id in workflow context", "sent": False}

    from core.db import async_session
    from core.models.channel import ChannelAccount
    from sqlalchemy import and_, select
    import uuid

    uid = uuid.UUID(str(owner_user_id))
    async with async_session() as session:
        result = await session.execute(
            select(ChannelAccount).where(
                and_(
                    ChannelAccount.user_id == uid,
                    ChannelAccount.channel == channel,
                    ChannelAccount.is_paired == True,
                )
            )
        )
        account = result.scalar_one_or_none()

    if not account:
        error_msg = f"No linked {channel} account for user {owner_user_id}"
        logger.warning("channel_output_no_account", channel=channel, user_id=str(owner_user_id))
        raise ValueError(error_msg)  # Caught by workflow executor, sets run failed

    from api.routes.channels import get_channel_gateway
    from channels.models import InternalMessage

    gateway = get_channel_gateway()
    msg = InternalMessage(
        direction="outbound",
        channel=channel,
        external_user_id=account.external_user_id,
        external_chat_id=account.external_user_id,  # DM: chat_id == user_id
        text=message,
    )
    await gateway.send_outbound(msg)

    logger.info(
        "channel_output_delivered",
        channel=channel,
        user_id=str(owner_user_id),
        external_chat_id=account.external_user_id,
    )
    return {"channel": channel, "message": message, "sent": True}
```

### Example 3: Keycloak Role Fetch

```python
# Source: backend/security/keycloak_client.py (to be created)
import httpx
import structlog
from core.config import get_settings

logger = structlog.get_logger(__name__)

async def fetch_user_realm_roles(user_id: str) -> list[str]:
    """
    Fetch a user's realm-level role names from Keycloak Admin API.

    Uses client_credentials grant with the backend's service account.
    Raises httpx.HTTPError if Keycloak is unreachable or returns an error.
    """
    settings = get_settings()
    base = f"{settings.keycloak_url}/realms/{settings.keycloak_realm}"
    token_url = f"{base}/protocol/openid-connect/token"

    verify = settings.keycloak_ca_cert or True

    async with httpx.AsyncClient(verify=verify, timeout=10.0) as client:
        # Step 1: Obtain admin access token
        token_resp = await client.post(token_url, data={
            "grant_type": "client_credentials",
            "client_id": settings.keycloak_client_id,
            "client_secret": settings.keycloak_client_secret,
        })
        token_resp.raise_for_status()
        admin_token = token_resp.json()["access_token"]

        # Step 2: Fetch realm role mappings
        roles_url = (
            f"{settings.keycloak_url}/admin/realms/{settings.keycloak_realm}"
            f"/users/{user_id}/role-mappings/realm"
        )
        roles_resp = await client.get(
            roles_url,
            headers={"Authorization": f"Bearer {admin_token}"},
        )
        roles_resp.raise_for_status()

        roles = [r["name"] for r in roles_resp.json()]
        logger.info("keycloak_roles_fetched", user_id=user_id, roles=roles)
        return roles
```

## Open Questions

1. **Keycloak service account permissions**
   - What we know: The Keycloak Admin API `GET /admin/realms/{realm}/users/{user-id}/role-mappings/realm` requires the calling service account to have `view-users` role in the `realm-management` client.
   - What's unclear: Whether `blitz-backend` already has this role assigned in the current Keycloak realm config.
   - Recommendation: Document the admin setup step. If the role is not assigned, the code will fail with 403 and fall through to the locked decision's behavior (fail the workflow run).

2. **Sub-agent mock data in Phase 3**
   - What we know: All three sub-agents (email, calendar, project) currently return hardcoded mock data -- they do not call real external APIs (Gmail, Google Calendar, CRM). This is by design: Phase 3 built the agent framework with mock data; real OAuth flows are deferred.
   - What's unclear: Nothing -- the agent_node will return real mock data formatted for channels, which is functionally complete for testing the wiring.
   - Recommendation: Wire the real sub-agent functions. The fact that they return mock data doesn't affect the wiring. When real OAuth is connected later, agent_node output will be real without any code changes in node_handlers.

3. **WorkflowState schema change**
   - What we know: Adding `workflow_name: str | None` to WorkflowState TypedDict is needed for the `[Workflow Name]` prefix on delivered messages.
   - What's unclear: Whether this counts as a "schema change" requiring version bump. It's a TypedDict addition (backward compatible) not a definition_json change.
   - Recommendation: Add it to WorkflowState. This is not a canvas definition schema change (no `schema_version` impact). It's a runtime graph state extension. TypedDict additions are backward compatible.
