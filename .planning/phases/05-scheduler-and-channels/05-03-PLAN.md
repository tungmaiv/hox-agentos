<objective>
Build the WhatsApp channel sidecar as an isolated Docker service that translates between WhatsApp Cloud API and InternalMessage format. The sidecar handles webhook verification (GET challenge-response), inbound message parsing, outbound text and interactive messages (up to 3 buttons), and message formatting (bold/italic only, strip code blocks).

Purpose: Enable users to interact with the Blitz agent via WhatsApp Business with text messaging and limited button support.

Output: WhatsApp sidecar FastAPI service, Cloud API wrapper, Dockerfile, Docker Compose service, and tests.
</objective>

  <action>
    Create the WhatsApp sidecar as a standalone Python project:

    1. **pyproject.toml**: project name "blitz-whatsapp-gateway", Python >=3.12, dependencies: fastapi, uvicorn[standard], httpx, structlog, pydantic>=2.0. Dev deps: pytest, pytest-asyncio, httpx.

    2. **WhatsAppAPI wrapper** (`whatsapp_api.py`):
       - Class with `__init__(self, access_token: str, phone_number_id: str)`
       - Base URL: `https://graph.facebook.com/v21.0/{phone_number_id}/messages`
       - `async send_text(to: str, text: str)` — POST text message
       - `async send_interactive(to: str, body_text: str, buttons: list[dict])` — POST interactive message with up to 3 buttons. Each button: {"type": "reply", "reply": {"id": action_id, "title": label[:20]}}. If more than 3 buttons, silently drop excess.
       - `strip_markdown(text: str)` — static method to convert markdown to WhatsApp-compatible format: keep *bold* and _italic_, strip code blocks (``` ... ```), convert [links](url) to plain "links (url)" text
       - All methods use httpx.AsyncClient with Bearer token auth header
       - Log errors with structlog, NEVER log the access token

    3. **FastAPI app** (`main.py`):
       - `GET /webhook`: WhatsApp verification challenge
         - Read hub.mode, hub.verify_token, hub.challenge from query params
         - If mode=="subscribe" and verify_token matches WHATSAPP_VERIFY_TOKEN env, return hub.challenge as plain text (Response with media_type="text/plain")
         - Otherwise return 403
       - `POST /webhook`: receive WhatsApp Cloud API event
         - Parse webhook payload: entry[0].changes[0].value.messages[0]
         - Extract: from (phone number) as external_user_id, wa_id, text.body as text
         - For interactive replies: extract button_reply.id as metadata.callback_data
         - Attachment detection: if message type is image/document/audio/video, respond "I can only process text messages"
         - Build InternalMessage (direction="inbound", channel="whatsapp", is_group=False always for MVP)
         - POST to BACKEND_URL/api/channels/incoming
         - Return 200 immediately (WhatsApp requires fast response)
       - `POST /send`: receive InternalMessage JSON
         - If actions present and len > 0: use send_interactive (max 3 buttons)
         - Otherwise: use send_text
         - Apply strip_markdown to text
         - Truncate at 4096 chars with "... (response truncated)"
       - `GET /health`: return {"status": "ok", "service": "whatsapp-gateway"}
       - Config from env: WHATSAPP_ACCESS_TOKEN, WHATSAPP_PHONE_NUMBER_ID, WHATSAPP_VERIFY_TOKEN, BACKEND_URL (default http://backend:8000)

    4. **Tests**:
       - `test_webhook.py`: test_verification_challenge_success, test_verification_challenge_wrong_token, test_text_message_forwarded, test_interactive_reply_forwarded, test_attachment_rejected
       - `test_send.py`: test_send_text_message, test_send_interactive_buttons, test_buttons_capped_at_3, test_markdown_stripped
       - Mock httpx calls to WhatsApp API and backend
