**Gap 1 fix -- middleware.ts (callbackUrl missing):**

In `frontend/src/middleware.ts` line 45, change:
```typescript
const token = await getToken({ req: request });
```
to:
```typescript
const token = await getToken({
  req: request,
  secret: process.env.NEXTAUTH_SECRET ?? process.env.AUTH_SECRET,
});
```

Root cause: In @auth/core 0.41.0 (used by next-auth 5.0.0-beta.30), `getToken()` does NOT auto-detect NEXTAUTH_SECRET from environment variables (unlike next-auth v4). When a stale session cookie exists, getToken() throws `MissingSecret` instead of returning null, which bypasses the middleware's callbackUrl redirect logic. Next.js error handling catches the throw and redirects to /login without callbackUrl.

**Gap 3+4 fix -- layout.tsx (AuthErrorToasts outside provider, no refetchInterval):**

In `frontend/src/app/layout.tsx`, restructure the body content so AuthErrorToasts is INSIDE SessionProvider (it is currently a sibling, placed after the closing tag). Also add `refetchInterval={300}` (300 seconds = 5 minutes) to SessionProvider for active polling:

```tsx
<body>
  <SessionProvider refetchOnWindowFocus={true} refetchInterval={300}>
    {children}
    <AuthErrorToasts />
  </SessionProvider>
</body>
```

The `refetchInterval={300}` means next-auth polls `/api/auth/session` every 5 minutes even while the tab is active. This catches cookie deletion / server-side session invalidation without requiring a tab switch. The 5-minute interval balances detection speed with network overhead for ~100 users.

**Gap 3+4 fix -- auth-error-toasts.tsx (only checks session.error, not unauthenticated status):**

Rewrite `frontend/src/components/auth-error-toasts.tsx` to detect BOTH scenarios:
- **Scenario A (existing):** session.error is "SessionExpired" or "RefreshAccessTokenError" -- token refresh failed but cookie still exists
- **Scenario B (new):** session transitions to null (status="unauthenticated") while on a protected page -- cookie was deleted or server-side session was invalidated

Updated component logic:

```tsx
"use client";
import { useEffect, useRef } from "react";
import { useSession, signOut } from "next-auth/react";
import { Toaster, toast } from "sonner";
import { usePathname } from "next/navigation";

export function AuthErrorToasts() {
  const { data: session, status } = useSession();
  const pathname = usePathname();
  const previousStatus = useRef(status);

  useEffect(() => {
    // Don't trigger on login page — would cause a redirect loop
    if (pathname === "/login") return;

    // Scenario A: session.error from failed token refresh
    if (
      session?.error === "SessionExpired" ||
      session?.error === "RefreshAccessTokenError"
    ) {
      toast.error("Your session has expired. Please sign in again.", {
        duration: 4000,
      });

      const timer = setTimeout(() => {
        void signOut({
          callbackUrl: `/login?callbackUrl=${encodeURIComponent(pathname)}&error=${session.error}`,
        });
      }, 1500);

      return () => clearTimeout(timer);
    }

    // Scenario B: session became null (cookie deleted, server-side invalidation)
    // Only trigger when transitioning FROM authenticated TO unauthenticated
    // (not on initial page load where status starts as "loading")
    if (
      status === "unauthenticated" &&
      previousStatus.current === "authenticated"
    ) {
      toast.error("Your session has expired. Please sign in again.", {
        duration: 4000,
      });

      const timer = setTimeout(() => {
        void signOut({
          callbackUrl: `/login?callbackUrl=${encodeURIComponent(pathname)}`,
        });
      }, 1500);

      return () => clearTimeout(timer);
    }

    previousStatus.current = status;
  }, [session?.error, status, pathname]);

  return <Toaster position="top-right" richColors />;
}
```

Key design points:
- `useRef(status)` tracks previous status to detect the authenticated-to-unauthenticated transition
- Only fires Scenario B when `previousStatus.current === "authenticated"` -- prevents false triggers on initial page load (where status starts as "loading" then becomes "authenticated")
- Both scenarios use the same toast message and redirect pattern for consistent UX
- `refetchInterval={300}` on SessionProvider (from layout.tsx change) ensures periodic polling catches server-side session invalidation even on an active tab
- `refetchOnWindowFocus={true}` (already present) handles the multi-tab case -- when user focuses a tab after signing out in another tab, session is re-fetched and Scenario B fires
