## 1. Session & Auth Hardening

### Problem

NextAuth v5 JWT cookies persist across host restarts. No `middleware.ts` protects routes. Client never acts on `session.error`. Expired sessions remain accessible.

### Solution (3 layers)

**Layer 1 — Next.js Middleware** (`frontend/src/middleware.ts`):
- Intercept all `/chat/*`, `/admin/*`, `/profile/*` routes
- Check for session cookie existence -> redirect to `/login` if missing
- Check `session.error` for `"SessionExpired"` or `"RefreshAccessTokenError"` -> redirect to `/login`
- Public routes whitelist: `/login`, `/`, `/api/auth/*`, `/api/copilotkit`

**Layer 2 — Backend Token Validation:**
- Chat page (server component) validates access token with backend on each load
- If 401 -> `signOut()` + redirect to `/login`
- Catches: backend restart with new secret, Keycloak token revocation, user disabled

**Layer 3 — Cookie & Session Config:**
- Explicit `maxAge` on session cookie matching token lifetime (8h local, dynamic for Keycloak)
- `httpOnly: true`, `secure: true` (production), `sameSite: "lax"`
- Client-side `SessionProvider` wrapper that watches `session.error` and auto-redirects

### Files

- NEW: `frontend/src/middleware.ts`
- MODIFY: `frontend/src/auth.ts` (cookie config)
- NEW: client-side session error handler component
- POSSIBLY NEW: `backend/api/routes/auth.py` -> `/api/auth/validate` endpoint

## 2. User Experience — Menu Redesign, Profile & Preferences

### Menu Redesign: Navigation Rail + Contextual Sidebar

Current problems:
- No global nav — each section builds its own isolated layout
- No way to reach Workflows from Chat
- Settings accessible only via gear icon in chat sidebar footer
- `AuthHeader` component exists but is unused
- Admin completely isolated with separate layout

**Solution: Navigation Rail (56px left edge)**

```
+------+------------------+----------------------------------+
| Rail | Context Sidebar   | Main Content                     |
|      | (conversations,   | (chat, canvas, settings form)    |
|      |  or settings      |                                  |
| Chat |  panels, etc.)    |                                  |
| Flow |                   |                                  |
| Gear |                   |                                  |
| Lock |                   |                                  |
|      |                   |                                  |
| ---- |                   |                                  |
| User |                   |                                  |
+------+------------------+----------------------------------+
```

**Navigation Rail items:**

| Icon | Label | Route | Visible To |
|------|-------|-------|------------|
| Chat | Chat | `/chat` | Everyone |
| Workflows | Workflows | `/workflows` | Everyone |
| Skills | Skills | `/skills` | Everyone |
| Settings | Settings | `/settings` | Everyone |
| Admin | Admin | `/admin` | admin/developer/it-admin roles |
| -- separator -- | | | |
| Avatar | Profile | `/profile` | Everyone (bottom-pinned) |

**Key behaviors:**
- Rail is always visible on authenticated pages (not on `/login`)
- Active item highlighted
- On mobile: rail collapses to a bottom tab bar (Chat, Workflows, Settings, Profile) — Admin accessible from profile menu
- Clicking avatar opens a dropdown: Profile, Sign Out
- Conversation sidebar only shows when on `/chat` — other sections use full width after the rail

### Profile Page (`/profile`)

- Display: avatar placeholder, name, email, auth provider badge (SSO / Local), roles
- Actions: Sign Out button, Change Password (local users only)
- Connected Accounts section: move from `/admin/credentials` for non-admin users
- Active Sessions: show current session expiry time, provider type

### User Preferences

Stored in new `user_preferences` table:

```sql
CREATE TABLE user_preferences (
    user_id     UUID PRIMARY KEY,  -- no FK (Keycloak users)
    preferences JSONB NOT NULL DEFAULT '{}',
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

JSONB schema: `{ "thinking_mode": bool, "response_style": "concise"|"detailed"|"conversational" }`

- **LLM Thinking Mode**: toggle — includes reasoning in responses vs direct answers
- **Response Style**: concise / detailed / conversational
- Loaded in `_master_node` (same as `user_instructions`) and injected into system prompt
- Backend: `GET/PUT /api/users/me/preferences`

**Design decision:** JSONB rather than columns — extensible without migrations for future preferences.

### Settings Page Reorganization

`/settings` keeps: Memory management, Chat preferences (rendering mode), Channel connections, Custom instructions.
Profile/account-related items move to `/profile`.

- NEW: `frontend/src/components/nav/navigation-rail.tsx`
- NEW: `frontend/src/app/(authenticated)/layout.tsx` — shared layout for all authenticated routes
- NEW: `frontend/src/app/profile/page.tsx`
- MODIFY: `frontend/src/app/chat/page.tsx` — use shared layout

## 3. Skill & Tool Platform (Phases A-E)

Reference: `docs/design/skill-tool-platform-proposal.md` for full technical details on A-D.

### Phase A — Agent Skills Standard Compliance

- Schema evolution: new columns on `skill_definitions` (license, compatibility, metadata_json, allowed_tools, source_url, source_hash, tags, category) + new `skill_assets` and `skill_dependencies` tables
- Enhanced SKILL.md importer: parse all standard fields, kebab-case name validation, source_hash tracking
- Skill exporter: SKILL.md + ZIP package output with MANIFEST.json for procedural skills
- ZIP bundle import: extract SKILL.md + MANIFEST.json + assets

### Phase B — Discovery & Catalog

- Skill catalog backend: PostgreSQL `to_tsvector` full-text search, category/tag filtering, pagination
- Tool catalog backend: search/filter tools by handler_type
- MCP Registry discovery: query `registry.modelcontextprotocol.io`, admin-only browse + install
- Skill catalog frontend: `/skills` page with search, filters, skill cards (integrated into nav rail)

### Phase C — Dependency & Security Hardening

- Dependency resolution: auto-extract deps from allowed_tools + procedure_json, validate on activation
- `allowed-tools` enforcement: restrict SkillExecutor tool calls to declared set
- Update checker: Celery periodic task re-fetches source_url, compares hash, creates pending_review version
- Enhanced SecurityScanner: new dependency risk factor (20%), data flow analysis factor (replaces author verification 10%)

### Phase D — Sharing & Collaboration

- Skill sharing between users via existing `artifact_permissions` system
- Export for external sharing: download SKILL.md or ZIP from UI
- On-premise marketplace: admin publishes approved skills, users browse and install

### Phase E — Enhanced Artifact Builder

**E-1: Executable Skill Generation**
- Builder generates complete `procedure_json` with proper steps, tool references, conditions, and prompt templates
- Builder generates `instruction_markdown` for instructional skills with proper Agent Skills format
- For tools: builder generates handler code scaffolding (Python function stub with Pydantic I/O models)
- Preview/dry-run mode: validate procedure_json against skill validator + show step-by-step execution preview

**E-2: Learn from External Repositories**
- Builder searches cached external repo indexes for similar skills based on user's description
- Shows top 3-5 relevant examples as reference
- "Fork" capability: select an existing external skill as starting point, builder pre-populates and adapts
- Builder prompt includes patterns from external skill ecosystem
- Output is always Agent Skills standards-compliant

**E-3: Security & Risk Gate on All Artifact Creation**
- Every artifact (built or imported) goes through SecurityScanner before activation
- New `security_review` node in builder LangGraph after `validate_and_present`
- SecurityReportCard A2UI component showing: trust score, factor breakdown, tool permissions, injection warnings, recommendation
- For `review` or `reject` recommendations: admin must explicitly approve
- Applies to all paths: builder, SKILL.md import, ZIP import, external repo import, fork/clone

**Enhanced SecurityScanner additions:**
- New factor: dependency risk (20%) — sandbox tools, admin tools, external MCP servers
- New factor: data flow analysis (replaces author verification 10%) — traces tool output->input chains, flags data exfiltration patterns
- Script file scanning for ZIP bundles: AST analysis of Python/JS scripts

### Security Invariants (Non-Negotiable)

1. Import quarantine — every imported skill goes through SecurityScanner -> pending_review
2. allowed-tools enforcement — executor restricts tool calls to declared set
3. Bundle scanning — ZIP imports scan all scripts for INJECTION_PATTERNS
4. MCP discovery is read-only — admin must manually approve + configure auth
5. Export redaction — strips credentials, user_ids, internal URLs
6. Dependency validation is local-only — no auto-install from external sources

### Phase E Files

- MODIFY: `agents/artifact_builder.py` — security_review node, external reference lookup, procedure_json generation
- MODIFY: `agents/artifact_builder_prompts.py` — prompts for executable skill generation with external examples
- NEW: `agents/artifact_builder_security.py` — security review node + A2UI SecurityReportCard
- MODIFY: `skills/security_scanner.py` — dependency risk + data flow factors
- MODIFY: `skills/importer.py` — invoke SecurityScanner on all import paths
- NEW: `frontend/src/components/a2ui/SecurityReportCard.tsx`

### B1: Embedding Sidecar Service (Expanded)

**Architecture:**

```
+---------+       +--------------+
| Backend |--HTTP--| Embedding    |  <- model pre-loaded on startup
| FastAPI |       | Sidecar :8003|
+---------+       +--------------+
                         ^
+---------+              |
| Celery  |--HTTP--------+
| Workers |
+---------+
```

**Configurable model:**
- Sidecar reads `EMBEDDING_MODEL` env var (default: `BAAI/bge-m3`) and `EMBEDDING_DIM` (default: `1024`)
- On startup: loads configured model, exposes `GET /health` with `{"model": "...", "dim": 1024, "status": "ready"}`
- Backend validates model info matches pgvector column dimension on startup

**Model change workflow:**
- Admin changes `EMBEDDING_MODEL` -> restart sidecar
- Backend detects dimension mismatch -> blocks embedding operations with clear error
- Admin endpoint: `POST /api/admin/memory/reindex` with `confirm=true`
  - Warning: "This will delete all existing memory vectors and re-embed from source text. Irreversible."
  - Celery task: truncate vector columns, re-embed all memory_facts and memory_episodes from stored text
- Audit log entry for model change + reindex

**Scale to 1000+ users:**
- Sidecar is stateless HTTP — horizontally scalable
- Batch API: `POST /embed` accepts `{"texts": ["...", "..."]}` — reduces HTTP overhead
- Response includes `model` and `dim` — callers validate on every call
- Backend uses `httpx.AsyncClient` with connection pool
- Future GPU acceleration: change sidecar Docker image only

**Files:**
- NEW: `infra/embedding-svc/` — FastAPI app, Dockerfile, requirements
- MODIFY: `docker-compose.yml` — add `embedding` service with health check
- MODIFY: `backend/memory/embeddings.py` — call sidecar HTTP instead of local FlagModel
- NEW: `backend/api/routes/admin_memory.py` — reindex endpoint
- NEW: `backend/scheduler/jobs/memory_reindex.py` — Celery reindex task
- MODIFY: `backend/core/config.py` — `EMBEDDING_SERVICE_URL` setting

**Expected outcome:** Pre-LLM overhead drops from 250-900ms to 50-150ms. DB sessions per request drop from 6-9 to 1-2.

## 5. Identity Configuration (Keycloak as Optional Runtime Config)

### Problem

Current stack requires Keycloak running, realm configured, clients created, and certs generated before first login. This is a significant deployment barrier. The Keycloak custom flat mapper also corrupts service account tokens.

### Solution

**Phase 1 — Make Keycloak optional at boot:**
- Backend boots with local auth only if no Keycloak config exists
- `security/jwt.py`: no Keycloak config -> skip RS256 validation, only handle HS256 local tokens
- JWKS fetching: only happens if Keycloak is configured
- Health check reports auth mode: `{"auth": "local-only"}` or `{"auth": "local+keycloak"}`
- Login page: only show "Sign in with SSO" button if Keycloak is enabled (check via `GET /api/auth/config`)

**Phase 2 — Admin Keycloak configuration UI:**
- New admin tab: **Identity** (or extend Config tab)
- Form fields: Issuer URL, Client ID, Client Secret, Realm name, CA cert path
- "Test Connection" button: validates config by fetching `.well-known/openid-configuration` and JWKS
- "Save & Enable" button: stores config to DB (`system_config` table), reloads JWKS cache
- "Disable SSO" button: reverts to local-only (with confirmation)

**Config resolution order:**

```
DB system_config (runtime)  ->  takes priority if set
      | fallback
ENV vars (KEYCLOAK_ISSUER, etc.)  ->  initial/default config
      | fallback
Not configured  ->  local auth only
```

**Data model** (in existing `system_config` table):

```json
{
  "key": "auth.keycloak",
  "value": {
    "enabled": true,
    "issuer": "https://keycloak.blitz.local:7443/realms/blitz-internal",
    "client_id": "blitz-portal",
    "client_secret": "***encrypted***",
    "realm": "blitz-internal",
    "ca_cert_path": "/path/to/cert"
  }
}
```

**Frontend handling:**
- NextAuth always defines both providers (Keycloak + Credentials) in `auth.ts`
- Login page calls `GET /api/auth/config` on mount -> conditionally renders SSO button
- If Keycloak not configured and user triggers SSO -> graceful error, redirect to login

**Also fixes:**
- Keycloak flat mapper issue (part of config validation)
- Moves from `admin/admin-cli` to proper service account credentials
- Absorbs TD-2 (secrets in docker-compose) — config moves to DB, env vars become optional fallbacks

### Files

- MODIFY: `backend/security/jwt.py` — conditional Keycloak validation
- MODIFY: `backend/security/keycloak_client.py` — runtime config from DB
- NEW: `backend/api/routes/admin_auth.py` — Keycloak config CRUD + test connection
- MODIFY: `backend/core/config.py` — DB-backed auth config resolution
